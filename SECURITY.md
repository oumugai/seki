# Security Policy

## サポートされているバージョン

seki は **pre-1.0** のプロトタイプ言語です。実 production 利用は **推奨しません**。

| バージョン | サポート状況 |
|---|---|
| 0.5.x | ✅ 最新、セキュリティ修正対象 |
| 0.4.x 以下 | ❌ サポート終了。0.5.x への upgrade を推奨 |

1.0 リリース以降は **マイナーバージョン単位で 1 年間** セキュリティ修正を
提供する方針に切り替えます。

## 脆弱性の報告

セキュリティ上の問題を発見した場合は、**公開 issue に投稿せず**、
プロジェクトオーナーへ直接連絡してください:

- メール: (プロジェクトオーナーに問い合わせください)
- GitHub のプライベートセキュリティアドバイザリ機能 (推奨)

24-72 時間以内に初動連絡を返します。修正がリリースされるまで脆弱性詳細の
公開を控えていただくよう協力をお願いします (responsible disclosure)。

## 既知の意図的な「安全でない」設計

以下は seki の **意図された設計** であり、セキュリティ脆弱性として
扱いません:

1. **`ffiLoad` / `ffiCallIntInt` / `ffiCallStrInt`** — 任意の `.so` を
   dlopen し、任意の関数を呼べます。これは **意図的な escape hatch** で、
   信頼できないコードを実行する環境では FFI ビルトインを使う seki プログラム
   を実行しないでください。
2. **`execShell` / `runCommand`** — 任意のシェルコマンド / プロセスを起動
   できます。同上の注意。
3. **`writeFile` / `removeFile` / `appendFile`** — 任意のパスにファイル
   操作が可能。サンドボックスは現時点で実装していません。
4. **`tcpListen` / `tcpConnect`** — 任意のポートでリッスン / 任意のホスト
   へ接続可能。
5. **依存型のサンプル検査** — 健全ではありません。型注釈をセキュリティ
   制約として扱わないでください (例: `def password : { s : String | safe s }`
   は型検査をすり抜ける可能性があります)。
6. **`Real` (= f64)** — IEEE-754 浮動小数点。`Real` 上の証明は
   厳密ではない (`by algebra` は Real を扱わない)。金銭計算には
   `Rat` (`Int × Pos`) を使用してください。

## サンドボックス化したい場合

現状: OS レベルでの隔離が必要です:

- Linux: `seccomp` プロファイル付きで実行
- コンテナ (Docker / Podman) でファイルシステム / ネットワーク を制限
- 別ユーザで実行

将来 Phase 9-10 で seki 内部のサンドボックス機能 (`--no-ffi`, `--no-exec`,
`--read-only` などの権限フラグ) を検討予定です。

## CVE 履歴

なし (現時点では発見されていません)。

## 暗号

seki には **暗号機能は含まれていません**。TLS や暗号ハッシュが必要な場合は:

- FFI 経由で OpenSSL / libsodium を呼ぶ
- Phase 9 で予定の `seki-tls` パッケージを待つ (rustls 経由)

自前で暗号プリミティブを実装しないでください — タイミング攻撃・サイド
チャネルへの耐性が保証できません。
